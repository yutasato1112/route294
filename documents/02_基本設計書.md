## 3. データ設計

### 3.1 データベース設計

本システムはDjango組み込みのUserモデルおよびSessionモデルを使用する。独自のデータベースモデルは定義しない。

#### 3.1.1 auth_user テーブル（Django組み込み）

| カラム名 | 型 | NULL | 説明 |
|----------|-----|------|------|
| id | INTEGER | NO | 主キー（自動採番） |
| username | VARCHAR(150) | NO | ユーザー名（一意） |
| password | VARCHAR(128) | NO | ハッシュ化パスワード |
| email | VARCHAR(254) | YES | メールアドレス |
| first_name | VARCHAR(150) | YES | 名 |
| last_name | VARCHAR(150) | YES | 姓 |
| is_active | BOOLEAN | NO | アカウント有効フラグ |
| is_staff | BOOLEAN | NO | スタッフ権限フラグ |
| is_superuser | BOOLEAN | NO | スーパーユーザー権限フラグ |
| date_joined | DATETIME | NO | アカウント作成日時 |
| last_login | DATETIME | YES | 最終ログイン日時 |

#### 3.1.2 django_session テーブル（Django組み込み）

| カラム名 | 型 | NULL | 説明 |
|----------|-----|------|------|
| session_key | VARCHAR(40) | NO | セッションキー（主キー） |
| session_data | TEXT | NO | シリアライズされたセッションデータ |
| expire_date | DATETIME | NO | セッション有効期限 |

### 3.2 ファイルベースデータ設計

#### 3.2.1 room_info.csv

| カラム | 型 | 説明 | 例 |
|--------|-----|------|-----|
| room_num | 整数 | 部屋番号（201〜1017） | 201, 302, 1017 |
| room_type | 文字列 | 部屋タイプコード | S, T |

**レコード数**: マスタデータ定義に依存（現行運用: 154件、2F〜10F）。room_info.csvの編集により部屋数・フロア構成を自由に変更可能。

**部屋番号体系**:
- 百の位: フロア番号（2〜10）
- 下2桁: フロア内通番（01〜17）
- 例: 201=2F-01号室、1017=10F-17号室

#### 3.2.2 times_by_type.csv

| カラム | 型 | 説明 | 例 |
|--------|-----|------|-----|
| type | 文字列 | 部屋タイプコード | S, T, bath, eco |
| time | 整数 | 清掃時間（分） | 24, 28, 50, 5 |
| label | 文字列 | 表示ラベル | シングル, ツイン |

**特殊タイプ**: bath（バス清掃）、eco（エコ清掃）は部屋タイプではなく清掃種別

#### 3.2.3 master_key.csv

| カラム | 型 | 説明 | 例 |
|--------|-----|------|-----|
| key | 文字列 | マスターキー名 | 2A, 3B, 10C |
| floor | 整数 | 対応フロア | 2, 3, 10 |

#### 3.2.4 weekly.csv

## 4. 機能設計

### 4.1 認証・認可設計

#### 4.1.1 認証フロー

```
ユーザー → [ログイン画面] → POST(username, password)
    ↓
Django authenticate() → ユーザー検証
    ↓ 成功                    ↓ 失敗
is_staff チェック             エラーメッセージ表示
    ↓ True    ↓ False         → ログイン画面へ戻る
login()     エラー表示
    ↓         → ログイン画面へ
redirect(/administrator/)
```

#### 4.1.2 権限モデル

| 権限レベル | フラグ | アクセス可能画面 |
|-----------|--------|-----------------|
| 一般ユーザー | is_staff=False | メイン、プレビュー、オートパイロット、Wincal、レポート、JSON保存 |
| スタッフ | is_staff=True | 上記 + 管理者パネル（CSV管理、ログ、メール設定、パスワード変更） |
| スーパーユーザー | is_superuser=True | 上記 + ユーザー管理（作成・削除） |

#### 4.1.3 カスタムデコレータ

```python
@staff_required  # @login_required + is_staff チェックの組み合わせ
# 未認証 → /login/ へリダイレクト
# is_staff=False → 403 PermissionDenied
```

### 4.2 メイン画面機能設計

#### 4.2.1 部屋グリッド表示

- **データソース**: room_info.csv + times_by_type.csv + master_key.csv
- **表示形式**: フロアごとの2Dテーブル（2F〜10F、逆順表示）
- **部屋ステータス**: 各セルに入力フィールド（ハウスキーパーID or 0 or 空）
- **色分け**: ステータスに応じたCSS適用

#### 4.2.2 データフロー（GET）

```
CSV読込 → room_info処理 → 2Dフロア配列生成
                              ↓
         times_by_type処理 → 部屋タイプ別時間設定
                              ↓
         master_key処理 → キー情報
                              ↓
         日付判定（18時以降は翌日）
                              ↓
         セッション確認（sidewind_flag）
            ↓ True                ↓ False
         allocation読込           初期表示
            ↓
         コンテキスト構築 → テンプレートレンダリング
```

#### 4.2.3 データフロー（POST - JSON読込）

```
JSONファイルアップロード → パース
    ↓
部屋割当データ展開 → room_inputs
    ↓
特殊部屋データ展開 → eco/ame/duvet
    ↓
ハウスキーパー情報展開 → house_data
    ↓
連絡事項展開 → remarks/contacts/spots
    ↓
コンテキスト構築 → テンプレートレンダリング
```

### 4.3 オートパイロット機能設計

#### 4.3.1 アルゴリズム概要

**手法**: 多目的最適化（200回のランダム戦略探索）

```
入力データ → バリデーション → 200回ループ
    ↓                            ↓
パラメータ解析              各戦略でアロケーション実行
    ↓                            ↓
ハウスキーパー配列構築      スコアリング（評価関数）
                                 ↓
                           最良解の選択
                                 ↓
                           ハウスキーパー番号再割当
                                 ↓
                           セッション保存 → リダイレクト
```

#### 4.3.2 制約条件

| 制約 | 種別 | 値 | スコアペナルティ |
|------|------|-----|----------------|
| 部屋クオータ | ハード | 指定数 | -10,000/部屋差 |
| ツインクオータ | ハード | 指定数 | -100/部屋差 |
| 通常フロア数 | ソフト | 2フロア以下 | -5,000/超過フロア |
| 全フロア数 | ハード | 3フロア以下 | -3,000/超過フロア |
| 3フロアペナルティ | ソフト | 1回 | -2,000 |
| フロア人数制限 | ハード | 3人/フロア | -800/超過人数 |
| エコ配分均等 | ソフト | 均等 | -500/差分 |
| ツイン配分均等 | ソフト | 均等 | -800/差分（差>2で-1,500追加） |

#### 4.3.3 割り当て戦略

| 戦略番号 | 内容 |
|----------|------|
| 0 | バス担当 + 通常担当（デフォルト順） |
| 1 | バス担当 + 通常担当（ツインクオータ降順） |
| 2 | バス担当 + 通常担当（ツインクオータ昇順） |
| 3 | バス担当 + 通常担当（部屋クオータ降順） |
| 4 | バス担当 + 通常担当（部屋クオータ昇順） |
| 5 | バス担当 + 通常担当（逆順） |
| 6-19 | バス担当 + 通常担当（シード付きシャッフル） |
| 20-39 | 上記0-19のフロア逆順処理版 |
| 40-199 | ランダムシャッフル |

### 4.5 管理者パネル機能設計

#### 4.5.1 AJAX通信設計

全ての管理操作はAJAX（非同期HTTP通信）で実行し、ページリロードを伴わない。

**リクエスト形式**:
```
POST /administrator/
Content-Type: application/x-www-form-urlencoded
X-Requested-With: XMLHttpRequest
X-CSRFToken: {token}

action={アクション名}&param1=value1&param2=value2
```

**レスポンス形式**:
```json
{"success": true, "message": "操作完了メッセージ"}
{"success": false, "message": "エラーメッセージ"}
```

#### 4.5.2 アクション一覧

| アクション | 権限 | パラメータ | 処理内容 |
